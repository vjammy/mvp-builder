# Business User Workflow

## Who This Is For

This guide is for founders, operators, product owners, and business-side collaborators who want to use MVP Builder without living inside the technical folders.

## The Main Idea

You do not need to understand every implementation detail.

Your job is to make sure the project is:

- clear
- scoped
- sequenced
- honestly verified

The technical builder or AI agent can do the execution work, but the business side still owns clarity, priorities, and acceptance.

## What You Should Open First

Start with:

1. `BUSINESS_USER_START_HERE.md`
2. `CURRENT_STATUS.md`
3. `STEP_BY_STEP_BUILD_GUIDE.md`
4. `COPY_PASTE_PROMPTS.md`
5. `MODULE_MAP.md`

These files exist so you do not need to read every folder.

## Your Responsibilities In The Method

As a business user, you are mainly responsible for:

- confirming the product goal
- confirming the target user
- protecting the MVP scope
- clarifying non-goals
- deciding what counts as success
- refusing to let vague work pass as complete

## The Six Business Stages

### 1. Decide

At this stage, you are answering:

- What are we actually building?
- Who is it for?
- What problem matters first?
- What absolutely must stay in v1?
- What is out of scope for now?

Do not move forward if the project still sounds like a brainstorm instead of a real first release.

### 2. Plan

At this stage, you are checking:

- whether the phase order makes sense
- where approvals or evidence are required
- what would block progress
- whether the technical plan is still appropriate for the MVP

Your main job here is to stop the team from pretending uncertainty is settled.

### 3. Design

At this stage, you are checking:

- whether the workflow actually makes sense for the user
- whether key states and edge cases are visible
- whether screens or steps are missing

You do not need to design every pixel. You do need to reject flows that only work on the happy path.

### 4. Build

At this stage, you are checking:

- whether the builder stayed inside the phase scope
- whether they changed the right things
- whether the handoff is understandable

Do not assume "the AI built it" means the phase is finished.

### 5. Test

At this stage, you are checking:

- whether tests were actually run
- whether evidence exists
- whether screenshots exist for UI work
- whether the recommendation is honestly earned

This is where many projects get sloppy. MVP Builder is intentionally strict here.

### 6. Handoff

At this stage, you are checking:

- whether the next person can understand the current truth
- whether risks and deferrals are visible
- whether the package is actually ready to continue

The handoff should be useful even if the original builder disappears.

## When You Should Stop The Process

Stop and ask for revision if:

- the brief is still vague
- scope keeps growing
- required acceptance criteria are missing
- tests were not really run
- evidence is weak or generic
- the docs say pass but the real state is unclear

## Questions A Business User Should Keep Asking

- If I removed the product name, would these docs still sound generic?
- Can I tell what is in scope versus deferred?
- Can I tell why the current phase passed?
- Could a new builder pick this up tomorrow without private chat history?
- Are we proving something real, or just producing more paperwork?

## How To Work With An AI Agent

The easiest pattern is:

1. Read the current status.
2. Copy the matching prompt from `COPY_PASTE_PROMPTS.md`.
3. Give the AI only the files required for the current stage.
4. Ask it to restate the goal, blockers, and next safe action.
5. Refuse to move on until evidence is real.

## What Success Looks Like

A strong MVP Builder workflow for a business user looks like this:

- the project is understandable in plain English
- the MVP is protected
- the builder knows what to do next
- every phase has a clear pass/fail decision
- the handoff is honest

That is the business value of the method.
